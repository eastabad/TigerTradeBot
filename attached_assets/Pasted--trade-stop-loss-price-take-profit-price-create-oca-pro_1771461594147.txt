• 然后 只有在 trade.stop_loss_price 或 take_profit_price 有值时，才会调用 create_oca_protection(...) 去真正下 OCA 订单。

如果这四个条件中任何一个不满足，这次即时 OCA 就根本不会尝试（连 “OCA protection created...” / “OCA creation failed...” 这类日志都不会出现）。

4. 精准定位：卡在什么条件上？

综合代码 + 数据 + 日志，可以精准锁定到这几点：

1. 即时 OCA 创建分支的入口是 tiger_status == 'filled' 之后的那段 routes.py 代码；
2. 进入这段之前，你已经：
  • 有 Trade（stop_loss_price / take_profit_price 均有值）
  • 有 OrderTracker（FILLED）
3. 但在这段里面：
  • ts_config.is_enabled 必须为 True 才会创建 TS+OCA；
  • 且 existing_ts 必须为 None（不能已有 active TS）；
4. 目前从日志看：
  • 没有 “Created trailing stop #… for TEM on immediate fill” 的日志
→ 要么当时 ts_config.is_enabled 为 False，要么当时已有 active TS，被 if not existing_ts: 短路；
  • 没有 “OCA protection created/failed for TEM on immediate fill” 的任何日志
→ 说明 那一整块 if trailing_pos: 里的 OCA 调用根本没跑过。

换句话说：

对于 TEM 这笔“刚刚测试”的订单，当时即时填单那一条 “TS + OCA” 分支实际上没有被执行；
之后你看到的 TrailingStopPosition #3，很可能是由 fallback/rebuild 或别的入口创建的，OCA 则尚未由任何路径触发。───
